This manual covers everything a GM or operator needs to run a
campaign: gaining GM privileges, injecting narrative, triggering
distillation, managing combat, and reading overseer telemetry.

> **Local play today.** The transport is being rebuilt (spec 043, #138): the old SSH
> flow is retired. Connect with the `aether-cli` over the loopback bypass (below);
> remote sign-in is coming (#138/#139).

---

## Gaining GM privileges

Connect from localhost and type `/gm`:

```bash
npm run play -w @garrettmanley/aether-cli -- <character> --insecure
/gm
```

Localhost connections are pre-approved and bypass OIDC (they are
the trust zone documented in
ADR-0005),
but are not auto-GM. You must opt in with `/gm` in each session.

## Injecting narrative

Use `inject.ts` to write directly to the ledger and broadcast to all
players without entering a player session:

```bash
npx ts-node inject.ts aether_frontier "A distant horn echoes across the valley."
```

The first argument is the campaign ID. The second is the narrative
text. The injection writes to the ledger and sends on the
`GLOBAL` scope.

## Managing the Lore Digest

The Chronicler distills ledger blocks into the Lore Digest
(`universe/lore/distilled_history.md`) automatically. To trigger a
manual distillation pass:

```bash
npx ts-node manual_distill.ts
```

Run this after a long session to ensure the Lore Digest reflects
recent events before the next session.

## Starting and ending combat

### Start an encounter

```bash
/gm startencounter Kira Vex skeleton-guard
```

The engine rolls initiative for every combatant, announces the turn
order, and enforces it for the remainder of the encounter.

### End an encounter

```bash
/gm endencounter
```

The engine broadcasts "Combat ends." and removes the initiative gate.

## Registering rules

Register a new rule (creature, weapon, spell, armor) at runtime:

```bash
npx ts-node inject.ts aether_frontier --register-rule creature path/to/creature.json
```

Or use the Rules Sage directly:

```bash
./core/target/release/core register-rule --rule-type creature \
  --rule-file universe/rules/seed/creatures/skeleton.json \
  --campaign aether_frontier \
  --requester-id <session-uuid>
```

Add permanent templates to `universe/rules/seed/` so they are
available on every server boot.

## Overriding difficulty class

To set a custom DC for the next skill check of a specific type:

```bash
npx ts-node inject.ts aether_frontier --dc-override stealth 18
```

The override applies to the next roll of that skill type and then clears.

## Reading overseer telemetry

The engine writes all `OVERSEER`-scoped events to `logs/overseer.jsonl`.
Each line is a JSON object. Use `maintenance.ts` to tail the log:

```bash
npx ts-node src/maintenance.ts
```

Common event types:

| Type | Meaning |
| --- | --- |
| `PLAYER_INPUT` | Raw input received from a player session |
| `CLASSIFIER_INTENT` | Classified intent (type + skill/weapon/spell) |
| `NPC_ACTION` | NPC reaction generated by ActorAgent |
| `ERROR` | LLM or Sage error during a DM cycle |
| `HALLUCINATED_NEXT_ACTOR` | LLM suggested an NPC not registered as a virtual session |
| `RULE_FETCH_FAILED` | A spell or weapon slug was not found in the registry |
| `BARE_NUMBER_INPUT` | A player typed a bare number (rejected) |

## Maintenance

### Free ports before relaunching

If the engine crashes, port 443 (the HTTPS gateway) may
remain bound. Free them:

```bash
npx ts-node src/maintenance.ts
```

### Rebuild the Rust core

Any change under `core/src/` requires a rebuild:

```bash
cd core && cargo build --release
```

## Character access management

Every character JSON under `universe/characters/` has two
optional access fields introduced by the #34 auth rework:

- `primary_owner`: a single `<provider-id>:<sub>` identity key
  (for example `google:1234567890`) identifying the player who owns the
  character.
- `operators[]`: additional allowed identity keys (delegates,
  co-players).

### Delegating access (`/gm delegate`, `/gm revoke`)

Since #37, GMs grant and revoke operator status from a GM
terminal—no JSON edits, no git commits. The mutations write to
`universe/characters/<name>.json` via the same atomic tmp+rename
path the access resolver uses, and emit `DELEGATE_GRANT` /
`DELEGATE_REVOKE` events to OVERSEER for audit (see #40).

Grant a permanent operator:

```text
/gm delegate gurga google:1234567890
```

Grant with a free-form note:

```text
/gm delegate gurga google:1234567890 --note "weekend co-GM"
```

Grant with a TTL (operator status expires automatically):

```text
/gm delegate gurga google:1234567890 --expires 2026-12-31T23:59:59Z
```

Revoke:

```text
/gm revoke gurga google:1234567890
```

**Identity-key format** is exactly `<provider-id>:<sub>`:
typically `google:<numeric-sub>`. Email-only inputs are rejected
because the access resolver matches on key, not email. To learn
an operator's key, ask them to attempt a login once and read the
`google:` value from the OVERSEER `ACCESS_DENIED` event in
`logs/overseer.jsonl` (or from the operator's `~/.config/...`
output during their first OIDC consent flow).

**Re-grants** are idempotent—running `/gm delegate` against an
existing operator updates the entry's `note` / `expires_at` /
`email` while preserving the original `added_at` timestamp. The
output reflects whether the row was added or updated.

**Self-revocation safety:** there's no built-in guard against a
GM revoking their own operator entry on a character they don't
own as primary. Avoid revoking yourself unless you have another
authorized identity ready to log back in.

See character-access-troubleshooting.md
for common tasks (reassigning ownership, handling self-claim
race conditions, force-releasing stuck sessions).
